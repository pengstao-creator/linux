## 二  uml类图

### 1 uml概述

#### 1.1 uml是什么

UML是**统一建模语言**（Unified Modeling Language）的缩写，它发表于1997年，是一个支持模型化和软件系统开发的图形化语言，为软件开发的所有阶段提供**模型化和可视化**支持。  - 多种类型图

1.2 特点

UML运用**统一的、标准化**的标记和定义组合了当今最好的面向对象软件建模方法。

#### 1.3 作用

UML对软件系统进行**面向对象**的**描述和建模**。

#### 1.4  UML与编程语言区别

UML是一门遵循精确语法的**图形语言**，与c语言，c++等不同，用来对一个系统建立模型。

UML既支持正向工程，也支持反向工程。

4)组合

**组合（****Composition****）关系也表示的是一种整体和部分的关系**，但是在组合关系中整体对象可以控制成员对象的生命周期，一旦整体对象不存在，成员对象也不存在，**整体对象和成员对象之间具有同生共死的关系**。

在UML中组合关系用带实心菱形的直线表示。

比如人的头（Head）和嘴巴（Mouth）、鼻子（Nose），嘴巴和鼻子是头的组成部分之一，一旦头没了，

嘴巴也没了，因此头和嘴巴、鼻子是组合关系，C++ 表示为：

5)依赖

**依赖（****Dependency****）关系是一种使用关系**，特定事物的改变有可能会影响到使用该事物的其他事物，在需

要表示一个事物**使用**另一个事物时使用依赖关系，大多数情况下依赖关系体现在某个类的方法使用另一个类

的对象作为参数。在UML中，依赖关系用带箭头的虚线表示，由依赖的一方指向被依赖的一方。

比如，驾驶员（Driver）开车，Driver类的drive()方法将车（Car）的对象作为一个参数传递，以便在drive()

方法中能够调用car的move()方法，且驾驶员的drive()方法依赖车的move()方法，因此也可以说Driver依赖

Car，C++代码为

```c++
class Car
{
    public:
    void move();
};
class Driver
{
    public:
    void drive(Car car)
    {
    car.move();
    }
};
```

​      继承，**关联(拥有)，依赖（使用）,聚合（整体部分），组合（整体部分，整体没了，部分也没了）**，。

从上文可以看出，关联关系、聚合关系和组合关系三者之间比较相似，本文的最后就来总结一下这三者之间

的区别。

关联和聚合的区别主要在于语义上：**关联的两个对象之间一般是平等的，聚合则一般是不平等的**。

聚合和组合的区别则在语义和实现上都有差别：**组合的两个对象之间生命周期有很大的关联，被组合的对象**

**在组合对象创建的同时或者创建之后创建，在组合对象销毁之前销毁，一般来说被组合对象不能脱离组合对**

**象独立存在，而且也只能属于一个组合对象；聚合则不一样，被聚合的对象可以属于多个聚合对象**。

再举例子来说：

你和你的朋友属于关联关系，因为你和你的朋友之间的关系是平等的，关联关系只是表示一下两个对象

之间的一种简单的联系而已，就像我有一个朋友

你和你借的书属于聚合关系，第一是因为书可以独立存在，第二是因为书不仅仅属于你，也可以属于别

人，只是暂时你拥有

你和你的心脏属于组合关系，因为你的心脏只是属于你的，不能脱离与你而存在

不过，实际应用中，我个人感觉三种关系其实没有区分得这么清楚，有些架构师甚至会说"组合和聚合没什么

区别"，所以，有时候不需要把细节扣得这么细，合理利用对象之间的关系给出设计方案即可。

### 2 单例模式（singleton）

#### 2.1 是什么&作用

​       类：一类事物抽象为类。  这类事物的特征抽象为属性。行为抽象为方法。

​       对象： 类实例化出来。  

=================

​    一个类确保它只有一个对象（实例）方案就叫做**单例设计模式。**      

  作用：不用重复的创建对象和释放对象。 节约了时间和效率。   **没有线程安全都用单例.**



​     **tcpSocket,request,response,requestResult,这些要用多例 .  剩余的都要用单例.**

#### 2.2 分类

   饿汉单例设计模式： 很饿，提前准备好，马上拿来使用。

   懒汉单例设计模式：最开始不创建，第一次使用的时候创建。

===============

​     构造函数私有化： 禁用拷贝构造函数    让其他类中不能建

​     自己创建一个静态实例。     饿汉单例  懒汉单例(如果长时间不用...用这个适合)

​    提供一个静态方法获取创建好的实例给别人就OK

#### 2.3 饿汉单例设计模式

```c++
#include <string>
#include <iostream>
using namespace std;
class Test
{
    private:
        string name;

            this->name=name;
```

### 3 工厂设计模式

工厂模式用于创建对象的实例，而不需要指定具体的类。可以分成工厂方法模式（一个产品一个工厂方法），简单工厂（一个工厂方法，但是要穿个类型进来），抽象工厂等

#### 3.1 工厂方法

```c++
#include <iostream>
#include <memory>

// 1. 抽象产品
class Sender {
public:
    virtual void Send() = 0;
    virtual ~Sender() {}
};

class MailSender : public Sender {
public:
    void Send() override { std::cout << "This is mail sender..." << std::endl; }
};

class SmsSender : public Sender {
public:
    void Send() override { std::cout << "This is sms sender..." << std::endl; }

class SendFactory {
public:

    virtual ~SendFactory() {}
};

// 3. 具体工厂 A (负责创建 MailSender)
class MailFactory : public SendFactory {
public:
    Sender* createSender() override {
        return new MailSender();
    }
};

// 4. 具体工厂 B (负责创建 SmsSender)
```

#### 3.3 抽象工厂

**抽象工厂模式 (Abstract Factory Pattern)** 的核心在于：**提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们具体的类。**

区别：

- **工厂方法模式**：针对的是**一个**产品等级结构（例如：只生产“发送器”，有邮件发送器、短信发送器）。它解决的是“怎么创建某一个具体产品”的问题。
- **抽象工厂模式**：针对的是**多个**产品等级结构组成的**产品族**（例如：既生产“发送器”，又生产“接收器”，还生产“过滤器”。而且这些产品必须配套，比如“邮件系列”包含邮件发送器+邮件接收器，“短信系列”包含短信发送器+短信接收器）。它解决的是“怎么创建一整套配套产品”的问题。

**生活类比**：

- **工厂方法**：你去一家店，只要买一把椅子。店家有木椅厂、铁椅厂。
- **抽象工厂**：你要装修整个房间，需要一套家具（椅子 + 桌子 + 沙发）。你需要“现代风格工厂”（产出现代椅+现代桌+现代沙发）或者“复古风格工厂”（产出复古椅+复古桌+复古沙发）。你不能混用（现代椅配复古桌通常不协调），抽象工厂保证你拿到的一整套风格是统一的。

```c++
#include <iostream>
#include <string>

// ================= 产品等级结构 1: 发送器 =================
class Sender {
public:
    virtual void Send() = 0;
    virtual ~Sender() {}
};

class MailSender : public Sender {
public:
    void Send() override { std::cout << "Sending via Email..." << std::endl; }
};

class SmsSender : public Sender {
public:
    void Send() override { std::cout << "Sending via SMS..." << std::endl; }
};

// ================= 产品等级结构 2: 接收器 (新增) =================
class Receiver {
public:
    virtual void Receive() = 0;
    virtual ~Receiver() {}
};

class MailReceiver : public Receiver {
public:
    void Receive() override { std::cout << "Receiving via Email..." << std::endl; }
};

class SmsReceiver : public Receiver {
public:
    void Receive() override { std::cout << "Receiving via SMS..." << std::endl; }
};

// ================= 抽象工厂 (核心) =================
// 负责创建一族产品：既能造 Sender，也能造 Receiver
class CommunicationFactory {
public:
    // 创建发送器
    virtual Sender* createSender() = 0;
    // 创建接收器
    virtual Receiver* createReceiver() = 0;
    
    virtual ~CommunicationFactory() {}
};

// ================= 具体工厂 1: 邮件系列工厂 =================
// 保证产出的都是邮件相关的组件
class MailFactory : public CommunicationFactory {
public:
    Sender* createSender() override {
        return new MailSender();
    }
    Receiver* createReceiver() override {
        return new MailReceiver();
    }
};

// ================= 具体工厂 2: 短信系列工厂 =================
// 保证产出的都是短信相关的组件
class SmsFactory : public CommunicationFactory {
public:
    Sender* createSender() override {
        return new SmsSender();
    }
    Receiver* createReceiver() override {
        return new SmsReceiver();
    }
};

// ================= 客户端使用 =================
int main() {
    // 1. 客户端决定使用哪个“系列” (产品族)
    // 如果这里换成 new MailFactory()，那么下面拿到的所有组件都会自动变成邮件系列的
    CommunicationFactory* factory = new SmsFactory(); 

    // 2. 工厂生产一整套配套产品
    Sender* sender = factory->createSender();
    Receiver* receiver = factory->createReceiver();

    // 3. 使用产品
    sender->Send();
    receiver->Receive();

    // 4. 清理资源
    delete sender;
    delete receiver;
    delete factory;

    return 0;
}
```

### 4 策略模式

**策略模式**是一种**行为型设计模式**。

它的核心思想是：**定义一系列算法（或行为），把它们一个个封装起来，并且使它们可相互替换。** 策略模式让算法的变化独立于使用算法的客户。

想象你要去旅行，目的地相同，但你可以选择不同的**交通策略**：

- 策略 A：坐飞机（快，贵）
- 策略 B：坐火车（中等速度，中等价格）
- 策略 C：骑自行车（慢，便宜，环保）

在代码中，如果不使用策略模式，你可能会写大量的 `if-else` 或 `switch` 语句来判断用户选了哪种交通方式。
使用了策略模式后，你把“飞机”、“火车”、“自行车”分别封装成独立的类。客户端只需要在运行时决定“注入”哪个策略对象，而不需要关心具体的实现细节。

核心角色

1. **抽象策略 (Strategy)**：定义所有支持的算法的公共接口（通常是接口或抽象类）。
2. **具体策略 (ConcreteStrategy)**：实现了抽象策略接口的具体类，每个类代表一种具体的算法。
3. **上下文 (Context)**：持有一个策略类的引用，负责调用策略对象执行算法。它不关心具体是哪个策略，只关心接口。

优点

- **符合开闭原则**：增加新策略（如新增“坐轮船”）只需添加一个新类，无需修改现有代码。
- **避免多重条件判断**：消除了大量的 `if-else` 或 `switch`。
- **算法复用**：具体的策略类可以在其他地方单独复用。

代码实现：

```c++
#include <iostream>
#include <string>
#include <memory>

// 1. 抽象策略 (Strategy)
class PaymentStrategy {
public:
    virtual void pay(double amount) = 0; // 统一的支付接口
    virtual ~PaymentStrategy() {}
};

// 具体策略 A: 支付宝支付
class Alipay : public PaymentStrategy {
public:
    void pay(double amount) override {
        std::cout << "Paid " << amount << " using Alipay." << std::endl;
        // 这里可以包含具体的支付宝 API 调用逻辑
    }
};

// 具体策略 B: 微信支付
class WechatPay : public PaymentStrategy {
public:
    void pay(double amount) override {
        std::cout << "Paid " << amount << " using WeChat Pay." << std::endl;
        // 这里可以包含具体的微信 API 调用逻辑
    }
};

// 具体策略 C: 信用卡支付
class CreditCard : public PaymentStrategy {
public:
    void pay(double amount) override {
        std::cout << "Paid " << amount << " using Credit Card." << std::endl;
        // 这里可能涉及验证 CVV 等逻辑
    }
};

//上下文持有策略的指针，并提供一个接口让客户端设置策略和执行操作。
// 上下文 (Context)
class ShoppingCart {
private:
    std::shared_ptr<PaymentStrategy> strategy; // 使用智能指针管理策略生命周期

public:
    // 设置或切换策略
    void setPaymentStrategy(std::shared_ptr<PaymentStrategy> newStrategy) {
        strategy = newStrategy;
    }

    // 执行策略
    void checkout(double amount) {
        if (!strategy) {
            std::cerr << "Error: No payment strategy selected!" << std::endl;
            return;
        }
        std::cout << "Processing checkout..." << std::endl;
        strategy->pay(amount); // 委托给具体策略执行
    }
};

int main() {
    ShoppingCart cart;
    double amount = 100.50;

    // 场景 1: 用户选择支付宝
    std::cout << "--- Scenario 1 ---" << std::endl;
    cart.setPaymentStrategy(std::make_shared<Alipay>());
    cart.checkout(amount);

    // 场景 2: 用户改变主意，切换为微信支付
    // 不需要修改 ShoppingCart 的代码，直接注入新策略即可
    std::cout << "\n--- Scenario 2 ---" << std::endl;
    cart.setPaymentStrategy(std::make_shared<WechatPay>());
    cart.checkout(amount);

    // 场景 3: 切换为信用卡
    std::cout << "\n--- Scenario 3 ---" << std::endl;
    cart.setPaymentStrategy(std::make_shared<CreditCard>());
    cart.checkout(amount);

    return 0;
}
```

### 1 什么是web服务器？

**Web 服务器**（Web Server）是指运行在服务器上的一种软件（或硬件+软件组合），它的主要职责是**监听网络请求**，**处理 HTTP/HTTPS 协议**，并向客户端（通常是浏览器）返回相应的资源（如 HTML 页面、图片、CSS、JS 文件等）或数据（如 JSON）。

核心工作流程

1. **监听**：服务器启动后，绑定到一个特定的 IP 地址和端口（通常是 80 或 443），等待客户端连接。
2. **接收**：当浏览器发起请求时，服务器通过 TCP 连接接收请求数据。
3. **解析**：服务器解析 HTTP 请求报文，提取出请求方法（GET/POST）、URL 路径、Header 等信息。
4. 处理：
   - 如果是**静态资源**（如 `.html`, `.jpg`），直接从磁盘读取文件。
   - 如果是**动态请求**（如 `.php`, `/api/user`），则调用后端程序或脚本生成内容。
5. **响应**：构造 HTTP 响应报文（包含状态码如 200/404、响应头、响应体），发送回客户端。
6. **关闭**：完成传输后，关闭连接（HTTP/1.1 默认保持连接，HTTP/2 多路复用）。

常见的 Web 服务器软件有：**Nginx**, **Apache**, **IIS**, **Tomcat**, **Caddy** 等。

本次就使用tcp协议做一个web服务器，可以用来简单的处理静态资源和动态请求。

### 2 会话类实现

**会话** 是指客户端和服务器之间**一次完整的、持续性的交互过程**。在 HTTP 服务器中，每个客户端连接就是一个会话。所以会话类中包含读取客户端数据，给客户端响应数据的操作。

**为了防止对象在异步操作完成前被意外销毁（导致程序崩溃），并允许对象在回调函数中安全地持有自己的“智能指针”。**HttpSession类必须要继承std::enable_shared_from_this<HttpSession>  例如：

1. 主线程或反应器线程调用 `session->do_read()`。
2. `do_read()` 注册了回调，然后返回。
3. 此时，外部的 `shared_ptr<HttpSession>` 可能已经离开作用域或被重置（引用计数归零）。
4. **对象被销毁**。
5. 几毫秒后，网络数据到达，Boost.Asio 尝试调用回调 lambda。
6. Lambda 里捕获的是 `this` (原始指针)，但**这个 this 指向的内存已经被释放了**。
7. **程序崩溃 (Segmentation Fault / Access Violation)**，因为你在访问野指针。

```c++
//std::shared_ptr 是智能指针，拷贝会增加引用计数
//使用 std::move 可以转移所有权，避免引用计数操作，性能更好
HttpSession::HttpSession(std::shared_ptr<tcp::socket> _socket, std::string webRoot)
	:_socket(std::move(_socket)),webRoot(std::move(webRoot)),
	buff(std::make_shared<boost::asio::streambuf>())
{
}

void HttpSession::do_read()
{
	// 检查 socket 是否有效
	if (!_socket || !_socket->is_open()) {
		std::cerr << "Socket is invalid or closed" << std::endl;
		return;
	}

	std::cout << "等待客户端发起请求" << std::endl;

	// 清空 buffer
	this->buff->consume(this->buff->size());

	// self 是一个 shared_ptr，指向当前对象
	// 当这个 lambda 执行时，self 仍然存在
	// 因为 lambda 持有 self 的拷贝，对象的引用计数至少为 1,所以对象不会被销毁
	auto self = shared_from_this();

	// 异步读取直到找到边界
	boost::asio::async_read_until(*this->_socket,*this->buff,"\r\n\r\n",
		[this,self](const boost::system::error_code& ec, size_t length) {
			// 收到客户请求，要对该请求做操作
			this->on_read(ec);
		});
}

void HttpSession::on_read(const boost::system::error_code& ec)
{
	// 解析请求参数
	HttpRequest req;
	if(!HttpParser::parse(*this->buff, req))
	{
		return;
	}

	std::string resp;
	 //根据请求参数做处理
	 //目前先只处理get请求
	if (req.method != "GET")
	{
		resp = HttpResponse::method_not_allowed();
	}
	else {
		// 目前的操作，就两种：html文本响应  json文本响应
		// 需要针对不同的操作类型，做不同的处理
		HandlerType type = Router::router(req);
		auto h = HandlerFactory::create(type, webRoot);
		resp = h->handle(req);
	}

	// 将响应结果做返回
	do_write(resp);
}

void HttpSession::do_write(const std::string& resp)
{
	auto self = shared_from_this();
	auto msg = std::make_shared<std::string>(resp);

	// lambda表达式中，捕获块中必须写[],因为msg是局部的，需要在 async_write 
	// 真正完成之后再销毁
	boost::asio::async_write(*_socket, boost::asio::buffer(*msg),
		[this, self,msg](const boost::system::error_code& ec, size_t) {
			if (!ec)
			{
				boost::system::error_code ignore_ec;
				_socket->shutdown(tcp::socket::shutdown_both, ignore_ec);
				_socket->close(ignore_ec);
				std::cout << "连接已关闭" << std::endl;
			}
		});
}
```

```c++
std::string HtmlHandler::handle(HttpRequest& req)
{
    //路径安全校验 防止目录遍历漏洞(如 ../../)
    //std::string::npos 表示"未找到"或"直到字符串结束"
    if (req.path.find("..") != std::string::npos)
    {
        // 找到了 ".."，说明存在目录遍历攻击
        return HttpResponse::not_found();
    }

    std::string path = req.path == "/" ? "/index.html" : req.path;
    std::string full = this->webRoot + path;

    auto content = FileUtil::read(full);
    if (content.empty()) return HttpResponse::not_found();

    auto mime = FileUtil::mime(full);
    return HttpResponse::ok(content, mime);
}
```

#### 3.3 工具类实现

FileUtil.hpp

用来读取静态文件内容

```c++
#pragma once
#include <string>
#include <fstream>

class FileUtil {
public:
    // 根据路径读取文件内容
    static std::string read(const std::string& path) {
        std::ifstream ifs(path, std::ios::binary);
        return { std::istreambuf_iterator<char>(ifs), {} };
    }

    // 根据文件类型，返回对应的响应格式
    static std::string mime(const std::string& path) {
        auto pos = path.find_last_of('.');
        if (pos == std::string::npos) return "application/octet-stream";
        auto ext = path.substr(pos + 1);
        if (ext == "html") return "text/html";
        if (ext == "css") return "text/css";
        if (ext == "js") return "application/javascript";
        if (ext == "json") return "application/json";
        if (ext == "png") return "image/png";
        if (ext == "jpg" || ext == "jpeg") return "image/jpeg";
        return "application/octet-stream";
    }
};
```

```c++
#include <iostream>

// 定义响应结果
class HttpResponse {
public:
    static std::string ok(const std::string& body, const std::string& content_type) {
        std::ostringstream oss;
        oss << "HTTP/1.1 200 OK\r\n";
        //oss << "Content-Type: " << content_type << "\r\n";
        oss << "Content-Type: " << content_type << "; charset=utf-8\r\n";
        oss << "Content-Length: " << body.size() << "\r\n";
        oss << "Connection: close\r\n\r\n";
        oss << body;
        return oss.str();
    }

    static std::string not_found() {
        std::string b = "<h1>404 Not Found</h1>";
        return ok(b, "text/html");
    }

    static std::string method_not_allowed() {
        std::string b = "<h1>405 Method Not Allowed</h1>";
        return ok(b, "text/html");
    }
};
```
